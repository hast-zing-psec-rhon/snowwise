# ADR 0002: Deploy the Rails app to Render with a managed PostgreSQL database

## Status

Accepted

## Context

The ski resort conditions app is a Ruby on Rails application using PostgreSQL and Rails 8 database-backed infrastructure components such as Solid Queue, Solid Cache, and Solid Cable. The app needs a low-friction hosted environment for an initial production deployment while preserving a path to more durable infrastructure later.

## Decision

Deploy the app to Render using a Git-backed Blueprint (`render.yaml`) with:

- one Docker-backed web service built from the repository `Dockerfile`;
- one managed PostgreSQL database;
- `DATABASE_URL` wired from the managed database into Rails;
- `SECRET_KEY_BASE` generated by Render;
- `PIRATE_WEATHER_API_KEY` configured as a non-synced secret in the Render Dashboard;
- Solid Queue running inside Puma for the initial single-service deployment.

## Rationale

A Render Blueprint keeps deployment configuration version-controlled and repeatable. A managed PostgreSQL database matches the app's production database adapter and supports Rails' database-backed cache, queue, cable, and application tables without introducing Redis or a separate worker service during the early product phase.

Running Solid Queue in Puma reduces operational surface area and cost for the first deployment. This is acceptable for a low-traffic initial app, but it trades off isolation: background jobs share memory and process lifecycle with the web server.

## Consequences

- The initial deployment is simple and inexpensive.
- Background job reliability and web latency are coupled to the web process.
- Render's free plans are suitable for early validation but may sleep, have resource limits, and are not appropriate for production traffic or time-sensitive scheduled weather ingestion.
- If weather ingestion becomes frequent or user traffic grows, split Solid Queue into a dedicated worker service and consider a paid database/web plan.

## Follow-up questions

- Should the app use a separate Render worker once scheduled weather refresh jobs are introduced?
- Should uploaded files remain local, or should Active Storage move to object storage before users can upload content?
- What uptime and freshness requirements should determine when to move off free-tier resources?
